## Arquitetura Técnica e Modularidade do Motor de Planeamento Financeiro

A arquitetura técnica do projeto `M6_G18_260601` representa um avanço significativo em relação à modelação financeira tradicional, frequentemente confinada a planilhas eletrónicas como o Excel [[30](https://www.scribd.com/document/489643803/Basic-financial-modelling-docx), [36](https://corporatefinanceinstitute.com/course/financial-modeling-guidelines/)]. A escolha de Python e JavaScript para implementar uma API RESTful com uma interface web associada não só demonstra competência técnica multifacetada, mas também posiciona o projeto numa linha de frente de modernização no domínio das finanças corporativas [[13](https://www.cfainstitute.org/programs/cfa-program/candidate-resources/practical-skills-modules/financial-modeling), [14](https://www.deltek.com/en/blog/financial-modeling)]. Esta abordagem modular e baseada em serviços promete maior escalabilidade, reutilização de código e facilidade de manutenção, características fundamentais para sistemas complexos de planeamento financeiro [[49](https://fminstitute.com/modeling-resources/financial-modeling-best-practices/)]. A análise da arquitetura revela tanto pontos fortes notáveis, como uma estrutura bem organizada e a dependência de ficheiros de configuração legíveis, como também lacunas críticas relacionadas com a robustez e a validação de dados, que são essenciais para a fiabilidade de qualquer sistema de apoio à decisão.

A principal força arquitetónica reside na clara separação de preocupações entre dados, lógica de negócio e apresentação. A estrutura do repositório, conforme descrito na documentação, organiza o código em diretórios distintos: `engine`, `inputs`, `modelo` e `projetos` [[9](https://mirror.nju.edu.cn/pypi/web/simple)]. Esta modularidade é uma prática recomendada em engenharia de software e modelação financeira, pois permite que cada componente seja desenvolvido, testado e modificado de forma independente, sem afetar o restante sistema [[15](https://www.sciencedirect.com/science/article/pii/S1877705815033366), [39](https://www.linkedin.com/pulse/validating-financial-models-colin-human-8bnxc)]. Os ficheiros de entrada, codificados em YAML (`inputs/`), servem como o ponto de origem para todas as premissas e cenários. A utilização de YAML é uma decisão estratégica inteligente; este formato de serialização de dados é humano-legível e fácil de gerir, permitindo uma definição clara e transparente das variáveis de input do modelo, o que facilita enormemente a auditoria e a adaptação do motor a diferentes casos de uso [[8](https://dev.to/jmarhee/example-of-yaml-generator-and-validator-in-python-1opk)]. A orquestração destes ficheiros de entrada é efetuada por um script principal, `run_model()`, localizado na pasta `engine/`. Este script atua como o cérebro do sistema, carregando as premissas, aplicando os cenários selecionados e invocando os módulos de modelação para calcular os resultados financeiros [[23](https://pypi.org/project/yamale/)].

O núcleo da lógica de negócio está encapsulado nos módulos dentro da pasta `modelo/`. Cada módulo tem uma função bem definida, o que contribui para a transparência e manutenibilidade do sistema. Por exemplo, o módulo `eoep.py` é responsável por gerar o calendário fiscal bottom-up para o ano de 2025, um processo crucial para a precisão contabilística das projeções [[12](https://www.ey.com/en_nl/services/finance-navigator/the-ultimate-guide-to-financial-modeling-for-startups)]. Outros módulos, como `monte_carlo_hub.py` para simulações de risco e `viabilidade.py` para cálculos de valorização, mostram uma divisão lógica das tarefas complexas em componentes menores e geríveis. A integração destes módulos através do script principal cria um fluxo de trabalho coerente, onde os dados brutos (YAML) são transformados progressivamente em outputs analíticos significativos (relatórios financeiros, KPIs, simulações). A separação entre a camada de lógica e a camada de apresentação é finalmente realizada pela API RESTful (servida por `main.py`) e pela interface web. A API expõe os resultados do modelo através de endpoints bem definidos, enquanto a interface web consome estes dados para visualização e interação, além de fornecer uma UI Swagger (`/docs`) para exploração da API, o que é uma excelente prática para desenvolvimento e depuração [[70](https://www.iso.org/obp/ui/en/#!iso:std:81291:en)].

No entanto, esta arquitetura sofisticada apresenta uma vulnerabilidade crítica: a ausência de um mecanismo robusto de validação de entradas. O motor depende inteiramente da correção sintática e semântica dos ficheiros YAML. Um erro simples, como uma vírgula mal colocada, um tipo de dado incorreto (por exemplo, um número onde se espera uma string) ou uma chave de dicionário ausente, pode levar a falhas catastróficas durante a execução do `run_model()`, resultando numa quebra total do pipeline de cálculo. Embora existam bibliotecas em Python como `yamale` que permitem validar documentos YAML contra um esquema definido, a sua implementação não foi identificada no projeto [[23](https://pypi.org/project/yamale/)]. A adição de uma camada de validação pré-execução seria uma melhoria de alto impacto, garantindo que apenas dados válidos e consistentes são processados, aumentando drasticamente a robustez do sistema [[7](https://stackoverflow.com/questions/3262569/validating-a-yaml-document-in-python)]. Sem ela, a usabilidade do motor fica condicionada à perfeição na escrita dos ficheiros de entrada, o que é uma exigência pouco realista para um utilizador final não-técnico.

Outra lacuna significativa é a ausência de testes automatizados, nomeadamente testes unitários. Num domínio tão sensível como o financeiro, onde pequenos erros podem ter consequências económicas substanciais, a garantia de qualidade através de testes é indispensável [[24](https://www.academia.edu/145076693/Integrated_software_testing_methodologies_for_financial_applications_requiring_high_reliability_standards)]. Funções críticas, como as que calculam o Valor Atual Líquido (VAL), a Taxa Interna de Retorno (TIR) ou executam a simulação de Monte Carlo (`viabilidade.py`, `monte_carlo_hub.py`), deveriam ser acompanhadas por testes unitários que verifiquem a sua corretude para entradas conhecidas e esperadas [[71](https://www.ibm.com/think/insights/unit-testing-best-practices)]. A implementação de testes, seguindo boas práticas como as delineadas na ISO/IEC/IEEE 29119-3, que oferece modelos de documentação para testes, seria um passo fundamental para garantir a integridade e fiabilidade dos resultados produzidos pelo motor [[72](https://www.iso.org/obp/ui/es/#!iso:std:79429:en)]. A falta de testes significa que qualquer alteração futura no código introduz um risco incontrolado de regressão, comprometendo a confiança no produto final.

A usabilidade do sistema, embora funcional, poderia ser aprimorada. A existência de uma interface web e de uma API documentada é um grande diferencial, mas as mensagens de erro devolvidas ao utilizador em caso de falha não parecem ser otimizadas para um público não-técnico. Em vez de dumps de stack trace em Python, seriam preferíveis mensagens de erro claras e concisas que indiquem o que correu mal e como o problema pode ser resolvido (ex: "Erro na linha 45 do ficheiro 'sales.yaml': o campo 'growth_rate' deve ser um número entre 0 e 0.2"). Além disso, a implementação de um painel de controlo visual para os indicadores-chave de desempenho (KPIs) e os resultados das simulações de Monte Carlo, em vez de depender exclusivamente da exportação para Excel, aumentaria imenso o valor comunicacional e analítico do projeto [[18](https://www.linkedin.com/pulse/advanced-financial-modeling-tools-tips-every-analyst-ekxof)].

Em síntese, a arquitetura técnica do `M6_G18_260601` é exemplar em termos de modularidade e visão sistémica, demonstrando uma compreensão profunda dos princípios de engenharia de software aplicados às finanças. O projeto possui todos os ingredientes para se tornar numa ferramenta de FP&A profissional: uma base de dados configurável, uma lógica financeira bem estruturada e uma camada de apresentação acessível. Contudo, para atingir esse potencial e satisfazer os critérios de uma avaliação académica rigorosa, a implementação de validação de entradas e testes unitários não é opcional, mas sim imperativa. Estas duas melhorias, combinadas com uma otimização da experiência de utilizador, transformariam um bom projeto técnico num excelente artefacto de investigação e desenvolvimento.

## Fundamentos da Lógica Financeira e Conformidade Normativa

A lógica financeira subjacente ao motor de planeamento do projeto `M6_G18_260601` é robusta e demonstra uma compreensão sólida dos conceitos fundamentais da avaliação de investimentos e da elaboração de demonstrações financeiras projetadas, alinhando-se com os referenciais curriculares da UC PEF, nomeadamente as Orientações de Empresa (OE4) e as Normas Contabilísticas de Relato Financeiro (NCRF) [[32](https://core.ac.uk/download/47136328.pdf), [50](https://iasplus.com/content/26533109-5f62-469d-8679-ac55de03a77a)]. A atenção aos pormenores técnicos, como a correta composição de taxas de juro e a aplicação do princípio bottom-up, eleva o projeto acima de uma modelação superficial, conferindo-lhe um grau de rigor e credibilidade que é essencial para a tomada de decisões empresariais informadas.

Um dos aspectos mais notáveis da lógica financeira é o tratamento cuidado das taxas de desconto. O modelo distingue explicitamente entre *spreads* de juro reais e a taxa nominal de desconto (WACC). A transformação de um *spread* real num valor nominal ocorre dinamicamente durante a execução do modelo através da fórmula $taxa\_nominal = (1 + inflação) \times (1 + spread\_real) − 1$. Este procedimento, conhecido como equação de Fisher, é um requisito padrão na análise de projetos de investimento em ambientes inflacionistas e demonstra um entendimento preciso da diferença entre rendibilidade nominal (a que se vê no papel) e rendibilidade real (a que se consegue em termos de poder de compra). A implementação correta desta relação é crucial para evitar subestimar ou sobrestimar o custo do capital e, consequentemente, o VAL do projeto. A consistência nesta abordagem, aplicada em múltiplas partes do modelo, reforça a confiança nos resultados de valorização.

A conformidade com as normas contabilísticas portuguesas é outro pilar da lógica financeira do projeto. A projeção do Estado de Resultados (DR) para o ano de 2025 baseia-se num princípio de modelação bottom-up, detalhado no módulo `eoep.py`. Este método, que constrói o resultado final a partir das operações individuais ao longo do tempo (mês a mês, neste caso), é considerado superior ao método top-down (que parte de uma previsão de resultado global e depois a decompõe) porque gera demonstrações financeiras mais realistas e fiáveis, refletindo melhor a dinâmica operacional da empresa [[12](https://www.ey.com/en_nl/services/finance-navigator/the-ultimate-guide-to-financial-modeling-for-startups), [14](https://www.deltek.com/en/blog/financial-modeling)]. Esta abordagem está alinhada com as NCRF, que definem a estrutura e o conteúdo das demonstrações financeiras [[51](https://pt.scribd.com/document/474344398/Os-Modelos-das-Demonstracoes-Financeiras-pdf)]. Além disso, a métrica de cobertura da dívida (DSCR), calculada no módulo `viabilidade.py`, parece estar de acordo com as recomendações do OE4 para a análise da capacidade de serviço da dívida, uma preocupação primordial para instituições bancárias como o CGD/BPI na avaliação de pedidos de financiamento [[54](https://www.scribd.com/document/851809220/SCALE-2-to-3-FINAL)]. A correta aplicação destas normas e diretrizes regulatórias confere legitimidade e credibilidade ao plano de financiamento do investimento no Hub Logístico.

A estrutura dos cenários financeiros (Base, Upside, Downside, Stress) é logicamente construída e cobre um espectro razoável de hipóteses futuras para a Grestel. As variações percentuais nos drivers financeiros-chave — volume de vendas, margem bruta (*spread* de preço) e custos fixos e semelhantes (FSE) — são controladas e justificáveis, permitindo uma análise de sensibilidade robusta. O cenário Base serve como ponto de referência, derivado diretamente dos ficheiros de entrada YAML. Os cenários Alternativo (otimista) e Pessimista (conservador) introduzem pressões controladas no mercado e nos custos, reflectindo um ambiente de negócios incerto. O cenário Stress, por seu lado, simula uma situação adversa, com uma queda inicial nas vendas, mas prevê uma subsequente recuperação, o que é realista para muitas empresas após um choque externo. A transparência na comunicação destas premissas, tal como feita no guia do projeto, é fundamental para a sua utilidade como ferramenta de tomada de decisão estratégica [[31](https://www.researchgate.net/publication/299969839_Sensitivity_Analysis_and_Scenarios)].

A lógica subjacente ao investimento no Hub Logístico é igualmente bem fundamentada. O CAPEX total de €6.000k é enquadrado num plano de financiamento estruturado, que combina dívida bancária (€4.500k a uma taxa de 4,15% com um período de carência até 2027) e fundos próprios/subsídios (€1.500k + €2.700k do programa PT2030). A atribuição de €2.700k como subsídio do PT2030, representando 45% do CAPEX total, é tratada de forma adequada, enquadrando-se nas Normas de Relato Fiscal Atrelado (NCRF 22) [[63](https://www.europarl.europa.eu/RegData/etudes/BRIE/2022/729408/EPRS_BRI%282022%29729408_EN.pdf)]. Esta distinção entre financiamento de natureza subvencionada e não subvencionada é crucial para a correta contabilização e para a análise da rentabilidade líquida do investimento. A libertação de capital de giro estimada em €2.500k (realizada em 2026 e 2027) e o crédito fiscal decorrente do investimento (RFAI), estimado em €600k, são incorporados no modelo e contribuem positivamente para o fluxo de caixa do projeto, demonstrando uma análise holística que vai além do simples investimento inicial [[2](https://cdn.qwenlm.ai/2208d912-5666-4c09-868f-663641f231fa/2e80c47c-5cc5-45d2-bf49-a0662166b15c_R&C24.md?key=eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9.eyJyZXNvdXJjZV91c2VyX2lkIjoiMjIwOGQ5MTItNTY2Ni00YzA5LTg2OGYtNjYzNjQxZjIzMWZhIiwicmVzb3VyY2VfaWQiOiIyZTgwYzQ3Yy01Y2M1LTQ1ZDItYmY0OS1hMDY2MjE2NmIxNWMiLCJyZXNvdXJjZV9jaGF0X2lkIjpudWxsfQ.GBDWgw7q4LAzo18oerCRSjwm6iKqiz5KKwjIus_W3LU)].

A lógica para o cálculo dos Fluxos de Caixa Livres (FCF), que constituem a base para todas as métricas de valorização, parece seguir a fórmula tradicional: $FCF = EBIT(1-t) + Depreciações - CAPEX - ΔNFM$ [[58](https://www.scribd.com/document/509496557/fin-4150-week-6-lectures)]. Este fluxo de caixa representa o dinheiro verdadeiramente disponível para todos os provedores de capital (acreedores e accionistas) após a empresa ter pago as suas dívidas e reinvestido no crescimento. A consistência na definição e cálculo do FCF é o alicerce sobre o qual a confiança no VAL e na TIR repousa. A análise económica do projeto é, assim, sustentada por uma lógica financeira coesa, rigorosa e conformada com as melhores práticas e normativas reconhecidas, tanto em nível académico como profissional e regulatório.

## Modelação e Projeção Financeira para o Período 2024–2029

A modelação e projeção financeira constitui o coração do motor de planeamento do projeto `M6_G18_260601`, cobrindo um horizonte temporal de seis anos (2024-2029) anualmente e um detalhe mensal para o ano de 2025. Este processo transforma as premissas financeiras e as decisões estratégicas, como o investimento no Hub Logístico, num conjunto coerente de demonstrações financeiras projetadas e indicadores de performance. A metodologia adotada, que combina projeções agregadas com uma modelação detalhada bottom-up para o primeiro ano, visa equilibrar a gestão do tempo de cálculo com a precisão necessária para uma análise aprofundada.

As projeções anuais para o período 2024-2029 são construídas a partir dos cenários definidos nos ficheiros de entrada YAML. Para o ano de 2025, a modelação é particularmente detalhada. O módulo `eoep.py` gera um calendário fiscal bottom-up, que serve de base para a projeção mensal do Estado de Resultados (DR) [[12](https://www.ey.com/en_nl/services/finance-navigator/the-ultimate-guide-to-financial-modeling-for-startups)]. Este método implica que os valores de receitas, custos e despesas são calculados mês a mês, acumulando-se para obter os valores anuais. Esta abordagem, embora computacionalmente mais intensiva, permite capturar efeitos sazonais, a dinâmica do ciclo de caixa e outras nuances operacionais que seriam perdidas numa projeção puramente anual. A precisão contabilística deste método é um forte ponto de partida para a construção de um modelo financeiro robusto.

Uma vez que o DR de 2025 está projetado, o resto do modelo prossegue de forma sequencial. Utilizando os valores projetados para o DR de 2025, o sistema constrói as projeções para o Balanço e para o Demonstrativo dos Fluxos de Caixa (DFC) para o mesmo ano. A ligação entre estas três demonstrações financeiras é mantida através da gestão do Capital Circulante Próprio e de Terceiros (NFM) e de outros postos do balanço, garantindo a consistência do modelo. As projeções para os anos subsequentes (2026-2029) são então construídas com base nestas premissas iniciais, aplicando as taxas de crescimento definidas nos cenários para as principais variáveis como o volume de vendas e os preços de venda. Esta abordagem híbrida — detalhada para o primeiro ano e agregada para os seguintes — é uma solução pragmática que equilibra o rigor e a gestão do tempo de computação.

Esta capacidade de responder a perguntas do tipo "what-if" é uma das funcionalidades mais valiosas de um motor de planeamento financeiro [[15](https://www.sciencedirect.com/science/article/pii/S1877705815033366), [49](https://fminstitute.com/modeling-resources/financial-modeling-best-practices/)]. Adicionalmente, o sistema calcula e disponibiliza um conjunto de indicadores de desempenho chave (IDCs), como o Valor Atual Líquido (VAL), a Taxa Interna de Retorno (TIR), o Payback e o DSCR, que são cruciais para a avaliação da viabilidade do investimento no Hub Logístico. A capacidade de exportar estes relatórios para formatos como o Excel facilita a sua análise e partilha com stakeholders não técnicos [[38](https://corporatefinanceinstitute.com/resources/financial-modeling/free-financial-modeling-guide/)]. A modelação e projeção financeira do `M6_G18_260601` é, portanto, um processo sistemático e bem estruturado, que transforma premissas estratégicas em análises financeiras quantitativas e acionáveis, cumprindo plenamente o objetivo de criar uma ferramenta de planeamento financeiro integrada.

## Análise de Viabilidade e Gestão de Risco do Investimento no Hub Logístico

A análise económica do projeto `M6_G18_260601` foca-se de forma central na viabilidade do investimento proposto no Hub Logístico da Grestel, um projeto de CAPEX de €6.000.000. A avaliação da viabilidade transcende a simples aplicação de métricas de valorização tradicionais, integrando uma abordagem sofisticada de gestão de risco através de simulações de Monte Carlo e análise de sensibilidade. Esta combinação permite não apenas determinar se o projeto é lucrativo, mas também compreender a probabilidade de sucesso e identificar os fatores de risco mais críticos que podem influenciar o resultado final.

O investimento no Hub Logístico é justificado por uma série de benefícios económicos tangíveis e intangíveis. A poupança operacional anual projetada é de €480.000, com um crescimento nominal de 4% ao ano, proveniente da otimização dos processos logísticos [[2](https://cdn.qwenlm.ai/2208d912-5666-4c09-868f-663641f231fa/2e80c47c-5cc5-45d2-bf49-a0662166b15c_R&C24.md?key=eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9.eyJyZXNvdXJjZV91c2VyX2lkIjoiMjIwOGQ5MTItNTY2Ni00YzA5LTg2OGYtNjYzNjQxZjIzMWZhIiwicmVzb3VyY2VfaWQiOiIyZTgwYzQ3Yy01Y2M1LTQ1ZDItYmY0OS1hMDY2MjE2NmIxNWMiLCJyZXNvdXJjZV9jaGF0X2lkIjpudWxsfQ.GBDWgw7q4LAzo18oerCRSjwm6iKqiz5KKwjIus_W3LU)]. Esta poupança, somada a uma redução adicional de €80.000/ano de quebras de inventário, contribui significativamente para a melhoria da margem de lucro operacional. Mais importante ainda, o hub é expectável gerar um volume de negócios incremental no canal B2C, estimado entre €500.000 e €1.150.000 por ano entre 2026 e 2029. Este aumento das vendas não só acelera a recuperação do investimento inicial, mas também diversifica a base de clientes da Grestel, alinhando-se com a estratégia da empresa de expandir a sua presença em e-commerce [[2](https://cdn.qwenlm.ai/2208d912-5666-4c09-868f-663641f231fa/2e80c47c-5cc5-45d2-bf49-a0662166b15c_R&C24.md?key=eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9.eyJyZXNvdXJjZV91c2VyX2lkIjoiMjIwOGQ5MTItNTY2Ni00YzA5LTg2OGYtNjYzNjQxZjIzMWZhIiwicmVzb3VyY2VfaWQiOiIyZTgwYzQ3Yy01Y2M1LTQ1ZDItYmY0OS1hMDY2MjE2NmIxNWMiLCJyZXNvdXJjZV9jaGF0X2lkIjpudWxsfQ.GBDWgw7q4LAzo18oerCRSjwm6iKqiz5KKwjIus_W3LU)]. O plano de financiamento do projeto é cuidadosamente estruturado, refletindo uma boa gestão da estrutura de capital. O investimento de €6.000.000 é financiado por um empréstimo bancário de €4.500.000 a uma taxa de juro fixa de 4,15%, com um período de carência de capital e juros até ao final de 2027 [[64](https://www.mdpi.com/1911-8074/16/2/116)]. Este período de carência permite que a empresa gere o investimento antes de assumir o serviço da dívida, mitigando o risco de liquidez no início do projeto. O restante financiamento, de €1.500.000, provém de fundos próprios. Uma componente vital deste financiamento é o subsídio do programa PT2030, no valor de €2.700.000, que representa 45% do CAPEX total. A correta contabilização deste subsídio, enquadrado nas Normas de Relato Fiscal Atrelado (NCRF 22), é crucial para a análise da rentabilidade efectiva do projeto [[63](https://www.europarl.europa.eu/RegData/etudes/BRIE/2022/729408/EPRS_BRI%282022%29729408_EN.pdf)]. A capacidade de serviço da dívida (DSCR) é um indicador chave monitorizado, e o modelo projeta que o DSCR médio do projeto será superior a 1,25, uma condição tipicamente exigida pelos bancos para garantir a segurança do empréstimo, em linha com as orientações do OE4 [[54](https://www.scribd.com/document/851809220/SCALE-2-to-3-FINAL)].

Para além das métricas de valorização, a análise de risco é um pilar fundamental do projeto. A implementação de uma simulação de Monte Carlo com 6 drivers financeiros-chave é uma característica de altíssimo valor. Em vez de apresentar um único resultado de VAL ou TIR, a simulação gera milhares de cenários possíveis, utilizando distribuições de probabilidade para variáveis como o volume de vendas, o *spread* de preço e o CAPEX [[56](https://www.researchgate.net/publication/270981194_The_Use_of_Monte_Carlo_Simulation_Technique_to_Support_Investment_Decisions), [66](https://www.researchgate.net/publication/226384052_Risk_analysis_in_investment_appraisal_based_on_the_Monte_Carlo_simulation)]. Para variáveis com limites claros, foram usadas distribuições triangulares, enquanto para outras, foram usadas distribuições normais truncadas. O resultado é uma distribuição de probabilidade para o VAL final do projeto. Com base nesta distribuição, o modelo pode calcular a probabilidade de o projeto ser efectivamente lucrativo, expressa como `P(VAL > 0)` [[22](https://www.researchgate.net/publication/387185685_Assessing_Geopolitical_Risks_and_Their_Economic_Impact_on_the_USA_Using_Data_Analytics)]. Esta abordagem, recomendada por consultorias como a PwC e a EY, transforma a análise de risco de uma discussão qualitativa para uma avaliação quantitativa e probabilística [[18](https://www.linkedin.com/pulse/advanced-financial-modeling-tools-tips-every-analyst-ekxof), [20](https://www.pwc.com/cz/cs/assets/Risk-management-and-Modelling-eBook-A4.pdf)]. A capacidade de responder a perguntas como "Qual a probabilidade de perdermos dinheiro neste projeto?" é um ganho de informação monumental para o gestor.

Complementarmente, a análise de sensibilidade tornado one-at-a-time ajuda a identificar os drivers de risco mais críticos. Esta análise mostra que o Volume de Vendas e a Taxa de Desconto (WACC) têm o impacto mais significativo sobre o VAL, sendo a sua direção de risco mais crítica. Uma diminuição no volume de vendas ou um aumento na taxa de desconto deterioram rapidamente a viabilidade do projeto. Por outro lado, fatores como o Spread de Preço e a obtenção de Subsídios/Incentivos têm um impacto positivo e significativo no VAL. Esta informação é vital, pois permite que a gestão da Grestel concentre os seus esforços de mitigação de risco e monitorização nos fatores que mais importam, optimizando o controlo sobre o destino do projeto. A combinação destas duas técnicas de análise de risco proporciona uma visão holística e matizada da viabilidade do investimento, indo muito além da simples análise de ponto de equilíbrio. O projeto `M6_G18_260601` demonstra, assim, uma compreensão profunda de que a tomada de decisão de investimento informada não se baseia apenas na resposta a uma única pergunta ("É este projeto rentável?"), mas sim na resposta a uma série de perguntas mais complexas ("Quão rentável é? Qual a probabilidade de o ser? E quais são as maiores ameaças e oportunidades?").

## Avaliação Crítica, Lacunas Identificadas e Recomendações de Melhoria

Apesar da excelência demonstrada na arquitetura técnica, na lógica financeira e na análise económica, o projeto `M6_G18_260601` apresenta algumas lacunas e áreas de melhoria que, se abordadas, poderiam elevar o seu rigor académico e o seu potencial como ferramenta de apoio à decisão. A avaliação crítica que se segue foca-se nessas áreas, formulando recomendações práticas e estruturando-as em prioridades de ação com base numa análise preliminar de impacto e esforço. Esta secção destina-se a fornecer aos docentes avaliadores um roteiro claro sobre os pontos fortes do projeto e as melhorias que o poderiam transformar de um trabalho de alta qualidade para um artefacto de ponta.

A primeira e mais crítica área de melhoria reside na **validação das premissas macroeconómicas e de valorização**. O modelo de valuation é extremamente sensível às premissas de inflação e de custo de capital (WACC). Embora estas premissas estejam definidas nos ficheiros de entrada YAML, a sua fonte e a justificação para os seus valores não são claramente documentadas no repositório. Para um projeto académico, a transparência na seleção de premissas é fundamental para a validação. A ausência desta informação impede uma avaliação completa da racionalidade subjacente aos cálculos. A recomendação é, portanto, a **Validação e Documentação das Premissas Críticas**. Este ajuste requer um baixo esforço — consiste em adicionar uma secção ao `guia_docentes.md` ou a um novo documento explicativo, referenciando as fontes para a taxa de inflação (ex: previsões do Banco de Portugal [[62](https://www.elibrary.imf.org/view/journals/002/2022/203/article-A001-en.xml)]) e para o WACC (ex: média ponderada dos custos do capital de empresas comparáveis do setor cerâmico, usando o modelo CAPM). O impacto desta melhoria é, no entanto, muito elevado, pois garante a credibilidade e a conformidade do modelo com os princípios de modelação financeira [[39](https://www.linkedin.com/pulse/validating-financial-models-colin-human-8bnxc), [48](https://www.linkedin.com/posts/peeyushchitlangia_20-years-of-building-financial-models-everything-activity-7448994241046994944-9-P1)].

A segunda área crítica é a **robustez técnica**, especificamente a **ausência de validação de entradas e de testes unitários**. Como já discutido, a dependência de ficheiros YAML não validados representa um risco grave de falhas imprevisíveis. A implementação de um validador de esquemas YAML (usando bibliotecas como `yamale` [[23](https://pypi.org/project/yamale/)]) antes da execução do modelo é uma melhoria de médio esforço com um impacto máximo na fiabilidade do sistema. Paralelamente, a implementação de **Testes Unitários** para os módulos financeiros (`viabilidade.py`, `monte_carlo_hub.py`) é uma prioridade de alta importância. Apesar de exigirem um esforço considerável para serem desenvolvidos e mantidos, os testes unitários são a única maneira de garantir a correção matemática dos cálculos financeiros, que são o núcleo do projeto [[24](https://www.academia.edu/145076693/Integrated_software_testing_methodologies_for_financial_applications_requiring_high_reliability_standards), [71](https://www.ibm.com/think/insights/unit-testing-best-practices)]. A criação de um conjunto de testes que cubra casos de teste conhecidos (ex: um projeto com fluxos de caixa simples cujo VAL/TIR seja facilmente calculável manualmente) seria uma demonstração de maturidade técnica e de rigor científico que distinguiria o projeto. A falta destes dois elementos representa a maior fraqueza do projeto em termos de engenharia de software.

A terceira área de melhoria foca-se na **transparência e usabilidade**, nomeadamente a **documentação insuficiente**. Embora existam guias, a falta de exemplos concretos de ficheiros YAML de entrada dificulta a reprodução dos resultados e a exploração do potencial do motor por parte de um avaliador ou de um utilizador futuro. A criação de **Exemplos de Cenários Customizados** na documentação, mostrando como modificar os ficheiros de entrada para explorar novas hipóteses, seria um pequeno esforço com um retorno em termos de usabilidade e pedagogia muito elevado. Da mesma forma, a elaboração de um **Glossário de Siglas e Termos** (explicando EOEP, NFM, CMVMC, etc.) aumentaria a acessibilidade do projeto para um público mais vasto, incluindo aqueles que não são especialistas em finanças corporativas. Estas melhorias, apesar de de menor impacto, contribuem significativamente para a qualidade geral da entrega final.
